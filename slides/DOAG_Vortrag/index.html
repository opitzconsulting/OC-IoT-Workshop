Evolution M2M to IoT

http://de.slideshare.net/zdshelby/coap-tutorial

  • M2M War vor IoT da. Gibt es schon seit Jahrzehnten. Aber:
  • M2M: Silos spezifische Protokolle, alles sehr spezialisiert, vor allem im industriellen Kontext. Jeder hat "sein M2M" Projekt gehabt, Systeme kommunizieren nicht mit außen, sind abgeschlossen
  • IoT + BigData: Daten werden über das Internet mittels "standardisierter" (mehr dazu später) Protokolle verschickt. BigData betreibt Analysen (über Systemgrenzen hinweg) , Things kommunizieren über das Web auch untereinander (zwischen System)

    CoAP als Beispiel

    http://de.slideshare.net/zdshelby/coap-tutorial

    Wir wollen über CoaP sprechen, weil es viele interessante Vorteile bietet.

    • UDP, RESTful, QoS auf Application Ebene
    • Wesentlich energieeffizienter als HTTP
    • Unnötige Roundtrips können vermieden werden
    • TLS auf Datagram Ebene

    • Unser Tipp: Graphenbasierte DB als Grundlage des Device Management im IoT

    Das IoT Netz ist ein Graph. Jeder Knoten ist ein Ding, alle Linien die Kommunikationswege. Knoten und Kanten können durch unstrukturierte Daten beschrieben werden.

    Das Netz ist wie ein Social-Network of things. Knoten sind Profile von Things, Kanten sind Verbindungen zwischen den Dingen (wie Freundschafen o.ä. aus Facebook)

    Ding A will Informationen zu C im Bereich N. Query an DB gibt mögliche Kommunikationspartner zurück. A kann dann Objekte aus Menge F nach

    • Welche Geräte / Kommunikationswege sind betroffen?

      Query an DB nach allen Kanten welche entsprechendes Muster besitzen

    • Welche Ausweichtechnologien gibt es?

      Für jedes Kommunikationspaar: Welche alternativen Kommunikationswege gibt es? Für alle Paare: Welche Alternativen sind am häufigsten?

    • Wie können die Geräte umgestellt werden?

      Für jeden betroffenen Knoten: Wie ist die Adresse (IPv6), wie wird neue Software eingespielt?

    Cluster

    http://camunda.org/features/

    Run the process engine on different cluster nodes on top of a shared database. Use this setup for scale up or high availability.The process engine supports heterogeneous clusters where different processes are executed on different nodes.

    Embedded

    http://camunda.org/features/

    Since the process engine is implemented in Java itself, it can be added as a library to any Java application. We call this the Embedded Process Engine. Alternatively the process engine can also be installed as a container service inside an Application Server or a lightweight container such as Apache Tomcat. This way the process engine can be used by multiple applications which can be redeployed without shutting down the process engine. <-- our approach (integrated in Spring)
