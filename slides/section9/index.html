Business Prozess Management (BPM)

Geschäftsprozesse

  • Geschäftsprozesse umspannen häufig mehrere Abteilungen
  • Sich häufig wiederholende Aktivitäten
  • Zweck/Ergebnisorientierung
  • häufig Wertschöpfend
  • Effizienz der Geschäftsprozesse wesentliches Erfolgskriterium
  • IT ist Vorraussetzung für Abbildung und Verwaltung

BPM

  • Die Verwaltung und Optimierung von bestehenden und neuen Prozessen
  • Eigene Prozesse müssen kontinuierlich verbessert werden um Mehrwert für Kunden zu schaffen
  • Bestehende Prozesse analysieren um Optimierungspotenzial zu entdecken

BPMN

  • Entstand als Reaktion auf BPEL, ein sehr technischer auf XML basierender Prozessautomatisierungsstandard
  • Ein Standard welcher ~verständlich~ Prozesse darstellt und dabei gleichzeitig von einer Prozessengine ausführbar ist
  • Prozesse können ineinander geschachtelt werden → mehrere Abstraktionsebenen
  • Eine Möglichkeit gesucht eine Struktur zu finden mit der Fach und IT Abteilung zusammen gebracht werden kann
  • Für verschiedene Zielgruppen verschieden präzise Darstellungen von Prozessen
  • Es werden 3 Level vorgeschlagen:
    1. Strategisch, logisch abstrakt
    2. operational process model. Muss noch nicht ausführbar sein aber semantisch korrekt
    3. Technisches Modell, welches dem Engineer an die hand gegeben wird

Camunda BPM

  • Open Source Framework für BPM
  • BPMN 2.0 process engine in Java
  • Sowohl Plattform als auch Modeller (Eclipse Plugin)

Standalone

http://camunda.org/features/

Use the process engine as a standalone service and interact with it from remote applications. Use the REST API for remote communication or implement your own access infrastructure using JMS or SOAP Webservices.

Cluster

http://camunda.org/features/

Run the process engine on different cluster nodes on top of a shared database. Use this setup for scale up or high availability.The process engine supports heterogeneous clusters where different processes are executed on different nodes.

Embedded

http://camunda.org/features/

Since the process engine is implemented in Java itself, it can be added as a library to any Java application. We call this the Embedded Process Engine. Alternatively the process engine can also be installed as a container service inside an Application Server or a lightweight container such as Apache Tomcat. This way the process engine can be used by multiple applications which can be redeployed without shutting down the process engine. <-- our approach (integrated in Spring)

Drools

rule "Switch OFF office_jack for jack"
            when
            u : User( username == "jack" )
            then
            HashSet<String> specials = new HashSet<String>();
            specials.add( "office_jack" );
            insert( specials );
            System.out.println("## drools: special rule to switch OFF office_jack for jack ##");
            end

Zeigt schon das Prinzip: Rule Engine Regeln erlauben es sich regelmäßig ändernde Zustände (wie Mitarbeiter wechselt Büro oder wird entlassen ...) ohne Neuimplementierung oder involvierung der IT Abteilung zu verändern.

Im Prototyp: Nur wenige Regeln, dennoch recht langsam. Rule Engines sind sinnig ab ~25 oder mehr komplexen Regelstrukturen. Bei wenigen rules ist der Overhead zu groß.
